Program Specification#

Control loop#

After users have launched your program, the program should:

  1. greet the users with the message "Welcome to the number to ASCII art generation program!", then
  2. keep asking users for input with the message "Enter a number that you want to convert to an ASCII art: ", then
  3. process the input:
    1. if users press <ctrl+d>, terminate the program and print "Thank you for using the ASCII art program!",
    2. if the input is valid (refer to input validation for details), print the ASCII art (refer to ASCII art patterns for details), then repeat from step 2
    3. otherwise, print an error message (refer to error messages for details) and ask for the input again with the message "Please enter a number: ", then repeat from step 3
tip

In some situations, the user will need to press ctrl-d twice in a row to end the program.

Input validation#

Your program should validate the user input in this order. A valid input means:

  1. It is not an empty input. e.g. "" is not a valid input.
  2. It only has one input. e.g. "0123" is a valid input, but "0123 4567" is not.
  3. A number contains only digits. e.g. "0123" is a valid input, but "o123" is not.

The checking must be in order; for example, the number of input checking should be performed before the digit checking. i.e. for input like "abc 4567", it should be caught by the second rule, not the third.

tip

  1. You can assume the input has at most 1024 characters. You don't need to validate the length of the input. i.e. you may specify a variable char input[1024]; for reading in user input.

  2. Instead of scanf("%s", cstring) to read input as a string separated by whitespaces, you can use scanf("%[^\n]%*c", cstring) to read input as a string separated by a newline character.

  3. In case of an empty input, you may need to do scanf("%*c"); to flush out the newline character remaining in the input buffer. Otherwise, you may enter an infinite loop.

  4. Alternatively, you may also use fgets to read a line from an input stream (e.g. stdin).

Error messages#

Error messages must be printed in a specific format. These formats are:

  1. For an empty input:
    Input is empty!
  2. For invalid numbers of input:
    "<input string>" contains more than one input.
    e.g. if the input is "0123 4567", this error message will be printed:
    "0123 4567" contains more than one input.
  3. For input that is not digits:
    "<input string>" is not a number.
    e.g. if the input is "abc", this error message will be printed:
    "abc" is not a number.

    Input number is "01234". String length is 5.

It contains 6 lines. Each line has 8 characters. Then you can print the ASCII art on screen using for loops (looping each line and each character and accessing the array with a proper index).

Git version control#

In the homework directory of cs220 (your local personal repository), you should create a new subdirectory named hw1. In that hw1 subdirectory, you will create your program in a new C source file named hw1.c. At the top of the file, be sure to add two comment lines that indicate your name and JHED ID (name-based).

Throughout your work on this assignment, be sure to frequently add, commit (supplying meaningful messages) and push your changes to your personal git repository. After you complete your work on the assignment, you will be asked to submit a gitlog.txt file, just as in the warmup homework. However, we expect your log for this homework to show more activities.

Recall that your code is always expected to compile without errors or warnings. Submissions that do not compile due to errors will earn zero points, so be sure to test your program on ugrad machines and submit to Gradescope early and often! And once you get a good start on the assignment, always have some earlier compiling version of your work pushed up to Github. Compiler warnings will result in grade deductions.

Submission Requirement#

Create a .zip file named hw1.zip which contains only hw1.c and gitlog.txt. Copy the hw1.zip file to your local machine and submit it as Coding Homework 1 on Gradescope.

Recall you can create your gitlog.txt file by running git log > gitlog.txt.

When you submit, Gradescope conducts a series of automatic tests. These tests do basic checks like making sure that you submitted the right files and that your .c file compiles properly. If you see error messages here (look for red), address them and resubmit.

No-compile Policy

Remember that if your final submitted code does not compile (has errors), you will earn a zero score for the assignment.

tip

You may re-submit any number of times prior to the deadline; only your latest submission will be graded.

info

Review the course syllabus for late submission policies (grace period and late days). You will want to save your late days for future assignments as they will be more involved.

You should also make sure that your code has a good style. You can look at the coding style guidelines here of a course you will take later that also applies to this course. In brief, you should make sure that your submission is well-formed:

  • it is not overcommented or undercommented
  • there are no ambiguous variable names
  • proper/consistent bracket placements and indentation
  • no global variables

Two notes regarding automatic checks for programming assignments:

  • Passing an automatic check is not itself worth points. (There might be a nominal, low point value like 0.01 associated with a check, but that will not count in the end.) The checks exist to help you and the graders find obvious errors. This will be true for most of the assignments; the actual grades are given manually by the graders, along with comments.
  • The automatic checks cover some of the requirements set out in the assignment, but not all. There will be hidden tests that test edge cases. In general, it is up to you to test your own work and ensure your programs satisfy all stated requirements. Passing all the automatic checks does not necessarily mean you will earn all the points.

Overview - Double-ended Queue#

In this assignment, you are going to implement a character double-ended queue (aka dQueue) and a templated version of it. Queue is a simple data structure that stores arbitrary objects and restricts their access to "first in, first out" (FIFO). It can be implemented using a linked-list that you have learned in class. dQueue is a variation of Queue such that the queue is open on both sides, which means it allows "first/last in, first/last out". A dQueue can be implemented using a doubly linked-list: a linked-list that has not only a pointer to the next node but also a pointer to the previous node. In this case it is common to store a pointer to the first node in the list (head) and one to the last node in the list (tail) within the class fields. Sometimes you'll see a double ended queue referred to as a deque.

This homework builds on in-class exercise 15, so you should make sure to understand and complete that before attempting this project!

This homework comes with three starter source files (DQueue.h, DQueueTest.cpp, and TDQueueTest.cpp). You should get these files from the public repository under homework/hw4. Make sure you have a copy of these files before you start writing any code.

Program Specification#

A Character Double-ended Queue#

To start this homework, check the comments and class definition of DQueue in DQueue.h, which is a dynamic dQueue of characters. Your task is to implement this class.

Below are a list of functions you need to implement and some tips for the implementation:

  1. Default constructor: this function should create an empty dQueue, i.e. both _head and _tail should be nullptr.

  2. Non-default constructor: this function takes a character input and creates a dQueue with one element. (Hint: to create a new _Node, you can use list-initialization, e.g. _Node({'a', nullptr, nullptr}).) In this case, the dQueue only has one node, so both _head and _tail should point to the same node.

  3. Copy constructor: applying the rule of 3, we have a non-trivial destructor; we should implement the copy constructor for a deep copy.

tip

For constructors, remember to make good use of member initializer lists.

You may want to implement operator<< or operator< first and reuse it to implement the copy constructor.

  1. operator=: applying the rule of 3, we have a non-trivial destructor; we should implement the assignment operator for a deep copy.
tip

When you implement your copy logic, draw it out on paper. That will help to figure out how to update the pointers (next and prev, _head and _tail etc.)

For the assignment operator, if you reallocate memory, make sure there is no memory leak.

You may want to implement operator<< or operator< first and reuse it to implement the assignment operator.

  1. Destructor: this function should deallocate all the dynamically allocated memory.

  2. operator<<: this overloaded operator takes a character and adds it to the tail of the dQueue. (i.e. a new _Node is allocated and the linkages at the tail are updated.)

  3. operator>>: this overloaded operator takes a character reference. It removes the last element in the dQueue (i.e. at the tail) and assigns it to the parameter reference. i.e. a _Node is deallocated. This function should not assign anything to the parameter character if the dQueue is empty.

  4. operator<: this overloaded operator takes a character and adds it to the head of the dQueue. (i.e. a new _Node is allocated and the linkages at the head are updated.)

  5. operator>: this overloaded operator takes a character reference. It removes the first element in the dQueue (i.e. at the head) and assigns it to the parameter reference. i.e. a _Node is deallocated. This function should not assign anything to the parameter character if the dQueue is empty.

tip

If the removal of an element makes the dQueue empty, you may need to update both _head and _tail pointers. The same is true if you insert a character into an empty list.

  1. operator==: this overloaded operator compares two dQueues and returns true if they contain the same characters, node by node; otherwise, returns false.

  2. operator+: this overloaded operator concatenates two dQueues and returns the concatenated dQueue. The conatenation is done by copying the implicit object, then adding a copy of the explicit parameter to the end.

tip

The operator+ assigns the result to a new variable, not to the implicit object. e.g. consider a = b + c. b + c is the same as b.operator+(c). The implicit object b remians unchanged after calling b.operator+(c), as does explicit parameter c. b.operator+(c) returns the concatenated result and it is then assigned to a in this example using the assignment operator.

You may want to implement operator<< or operator< first and reuse it to implement the operator+.

  1. std::stream& operator<<: this overloaded operator prints a dQueue in this format: <data1> -> <data2> -> ... -> <datax>\n. For example, if we have queue having three elements a, b, and c. This function should print this string: a -> b -> c\n. If the dQueue is empty, print empty queue\n.

You are given a file named DQueueTest.cpp that provide basic tests for your implementation. Once you are done implementing DQueue.cpp, you should compile and run DQueueTest to make sure that all test cases pass. Keep in mind that these are minimal cases. You should develop your own test cases and test your program thoroughly.

A Templated Double-ended Queue#

Next, you will generalize the DQueue from above to create a templated double-ended queue. Note that you will write your own header file for this part. Your templated class must be called TDQueue and be declared in TDQueue.h and fully defined (i.e. implementation of functions longer than one line should occur) in a file named TDQueue.inc. Note that the file has the .inc extension but not .cpp. The TDQueue.inc should be included at the end of the TDQueue.h file after the class declaration (this is because we cannot split templated classes into fully separate .h and .cpp files). Also, note that the TDQueue class must be templated so that it works for various types as applicable.

Remember to keep data private, and const to protect objects (explicit and implicit parameters) whenever you can. Your TDQueue class must contain the same public functions and overloaded operators as the DQueue class, except that parameters for the data stored will now be of some generic type (a template parameter) instead of a char.

Your template specification may look like this:

template< typename DataType >

You are given a file named TDQueueTest.cpp that tests your implementation (for the templated version). Once you are done implementing TDQueue.h and TDQueue.inc, you should compile and run TDQueueTest to make sure that all test cases pass. Also, keep in mind that these are minimal cases. You should develop your own test cases and test your program thoroughly.

Other Requirements#

  • Do not eliminate or change anything declared in the DQueue.h file. You can add more private (helper) functions if you want to, but your DQueue.cpp file should at least provide an implementation for all the functions that are declared in DQueue.h file.
  • Your program should only use new and delete for dynamic memory allocation/deallocation. Do not make use of C functions such as malloc, calloc, free, etc., for that purpose.
  • All variables must be declared inside functions. No variables should be global or extern.
  • Do not use auto.
  • Use header guards in all header (.h) files.
  • Do not use using in .h or .inc files.
  • In C++ source files (.cpp), you may import individual symbols using statements like using std::string. Do not use using namespace <id>, either in headers or in source files.
  • Make use of gdb to debug and run valgrind to ensure no memory leakage or invalid memory usage regularly. Your submitted program should not have a memory leak problem.
  • Do develop your own test cases and test your functions thoroughly. You may share your test cases with your classmates, but never your source code.

Submission Requirements#

Makefile and Compiling#

Create and submit a Makefile with two rules (aside, perhaps, form the clean rule):

  1. a rule that creates a target executable called DQueueTest for the character double-ended queue test, and
  2. a rule that creates a target executable called TDQueueTest for the templated double-ended queue test.

These two targets should compile with no errors or warnings using the typical C++ compilation command: g++ <source> -Wall -Wextra -std=c++11 -pedantic. Do not use any special libraries that require different or additional compiler options.

Git Log and README#

In the assignment's folder of your private repository (not the team repository you made for the midterm project), create a new subfolder named hw4. Do your work in that subfolder and use git add, git commit, and git push regularly. Use git commit and the associated comments to document your work. e.g. if you just modified DQueue.cpp to add the constructor, you might do git add DQueue.cpp; git commit -m "DQueue constructor is added!"; git push.

Your submission includes a copy of the output of git log showing at least four commits to the repository. Save the git log output into a file called gitlog.txt (e.g. by doing git log > gitlog.txt).

Also, please submit a file called README (not README.txt or README.md, etc - just README) including information about what extra helper functions you added (if any) and anything the graders should know about your submission. In README you should:

  • Write your name and JHED ID at the top of the file
  • Briefly justify the structure of your program; why you defined the functions you added, etc
  • If applicable, highlight anything you did that was particularly clever
  • If applicable, tell the graders if you could not do everything. Where did you stop? What did you get stuck on? What are parts you already know do not work according to the requirements?

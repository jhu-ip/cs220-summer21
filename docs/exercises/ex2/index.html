Day 2 In-class Exercise

info

This is an in-class exercise. An exercise page like this one will contain a brief description but is intended to be supplemented by discussion during our meeting time. Complete the exercise to the best of your ability. Feel free to talk with other students as you work, and do not be afraid to ask questions if you get stuck. Aim to complete as much as possible during our meeting, and try to finish on your own before our next class meeting. Remember to submit on Gradescope once or more, checking the autograder results. For credit, you must submit by the end of the semester.

Learning Objectives#

Objectives

This exercise should help you gain familiarity with

  • Unix/Linux commands
  • Remote access - ssh
  • a text editor - emacs (or vim)
  • the distributed version control tool - git
  • a tool for bundling files - zip
  • a tool for transferring files - scp
  • C control flow

Part 1 - Remote access#

Introduction#

As mentioned in the last exercise, you will be working remotely on CS Ugrad machines. Now, you should have your CS accounts ready. It's time to get familiar with the remote access tool - ssh.

Instructions#

  1. Open a terminal on a Linux/Mac machine or Windows WSL.

    tip

    If you forget how to open a terminal, revisit the last exercise.

  2. Use this command to remote access a server:

    ssh USERNAME@SERVER_NAME

    where USERNAME is your CS account user name and SERVER_NAME is ugradx.cs.jhu.edu, or any machine ugrad1.cs.jhu.edu, ugrad2.cs.jhu.edu, â€¦ ugrad24.cs.jhu.edu.

  3. You will be asked to type in your password. Once you input the correct password, you should see a message like the below:

    You should also notice that the @MACHINE_NAME has changed from your local machine to the remote server. In the above example, it changed from my laptop MSI_SINGCHUN to ugradx. This is a hint that tells you if you are on your local machine or the remote server.

    If you hear us ask you to "ssh to ugrad", it implies that you should execute this command and work remotely on ugrad instead of working on your local PC.

    For Windows users

    Windows users can ssh directly using Windows Powershell. However, you are advised to download and use PUTTY. Directly "sshing" into ugrad machines via Windows Powershell has caused an issue with keyboard layouts in the past!

  4. (Optional) You should come back to this after finishing part 3. You have been using your username and password to ssh. To skip typing password every time you ssh, you may set up ssh key.

    1. Follow the instruction here to (only the first step) create your own ssh public/private key on the ugrad server.
    2. Now, on the ugrad server, copy id_rsa.pub to authorized_keys. You may use this command:
      cp ~/.ssh/id_rsa.pub ~/.ssh/authorized_keys
      authorized_keys tells ssh which public key is allowed to remote access.
    3. Then, go back to your local machine and use the scp command in Part 3 to copy ~/.ssh/id_rsa from the ugrad server to ~/.ssh/ on your local machine. You should use -p to preserve the modes of the file. For example:
      scp -p USERNAME@ugradx.cs.jhu.edu:~/.ssh/id_rsa ~/.ssh/
      Now, when you ssh on your local machine, ssh will use the private key id_rsa to establish the connection. For a valid public/private pair, you will be granted access from your local machine to the ugrad server via ssh keys. From now on, you no longer need to type your password when "sshing".

Checkpoints#

  1. How can you remote access an ugrad server using ssh?
  2. Looking at your terminal, can you immediately tell if you are on your local PC or an ugrad server?

Part 3 - File transfers between local and remote machines#

Introduction#

You are work remotely on the ugrad machine, but you may work locally on a local machine, and you also need to get the files for homework submission. You need a tool to transfer files between local and remote machines. scp is the right tool for this purpose. In this part, you will practice how to transfer files between local and remote machines, bundle files and directories into a zip, and refresh your Linux commands.

Instructions#

  1. If you are not on a ugrad server, ssh to one.

  2. Create a directory called test and a file called temp under your home directory. You can create an empty file by using the touch command. e.g. touch temp.

  3. Change into test directory, and create two files called hello and world.

  4. Go back to your home directory.

  5. Bundle the directory test and file temp running this command: zip -r bundled.zip test temp. Here -r means recursively. It is required to bundle files inside test.

  6. Verify what's in bundled.zip by running this command: unzip -l bundled.zip. You should see test, temp, test/hello and test/world.

  7. Go back to your local machine and copy bundled.zip from the remote machine to the current directory on your local machine. Our tools for accomplishing this will be scp (Mac or Linux or Windows Powershell), or either pscp or WinSCP if installed on Windows.

    Mac or Linux Users

    In a terminal on your local computer, copy your .zip file from ugrad to your local computer using scp <source> <destination>. For example, open a new Terminal window on your Mac (not the one in which you are connected to ugradx!), navigate to the folder into which you want the file to be copied and type: scp USERNAME@SERVER_NAME:PATH_TO_FILE .. The source argument above first indicates in what account and on what machine you want to look, then, after the colon, it indicates a relative path from your home directory. The destination argument, however, is simply a dot, because you are typing this command on your Mac and want the file to be transferred to the folder on your Mac that you are currently working in. So, there is no need to specify the destination machine; it is the local one by default.

    Tip

    Do not try to type a ./ after the colon to start the relative path name, or a ~/ to indicate an absolute path name; scp will not understand that notation.

    Windows Users

    You will need pscp here, which should have been installed already automatically when you installed Putty. Copy your .zip file from ugrad to your local computer using pscp <source> <destination>. For example, you could open a Command Prompt window or Powrshell and type pscp USERNAME@SERVER_NAME:PATH_TO_FILE .. For Windows where WinSCP has been installed: connect to ugradx using the WinSCP interface and drag the file from the ugradx side to your local computer's side.

  8. On your local machine, use the software you like to unzip it and verify if you have test, temp, test/hello and test/world.

  9. Modify the test files as you like, and zip them back as bundled.zip. Now copy it back to the ugrad server. e.g. by running: scp bundled.zip USERNAME@SERVER_NAME:~/.

  10. At last, ssh to ugrad and unzip the file by running unzip -o bundled.zip. Note that -o means override the existing files. Use it with caution! Then, cat the file (e.g. cat test). You should see the changes you made.

Checkpoints#

  1. How do you transfer files from remote to local machine?
  2. How do you transfer files from local to remote machine?
  3. How do you zip files?
  4. How do you verify zipped files?

Part 4 - Git#

Introduction#

Version control is an important aspect of software development. You will practice using git for your assignment version control and late team collaboration in this course. In this part, you will get familiarity with git and establish the concepts of local and remote repositories.

Instructions#

Clone a remote repository#

  1. You should have received (and accepted) an invitation email to join the course Github organization. Login and go to the course Github organization page here, you should be able to see a personal repository with the name 2021-summer-student-JHED where "JHED" is your actual JHED name. This will serve as your private space from here on that you will use for your exercises and homework. Only you and the course staff have access to your private repo.

    info

    If you don't have access to the repository, please make sure you have accepted the invitation to the jhu-ip organization. If you haven't received an invitation, please contact your instructor.

    If you are on your personal Github page, you probably need to change to the course organization (i.e. jhu-ip) on the left menu as follows:

    where instead of test, you should see your own Github user name.

    Besides your own personal repository, you should also see cs220-summer21-public - our course repository, where you can find the starter codes for the exercises, homework and projects. This is a remote repository on Github, and it is visualized using the web interface. With the web interface, you can navigate what's in the repository as if you navigate different directories/files.

  2. A repository is essentially structured files and directories. We can also visualize it on our local machine with the OS's file system so that we can modify the directories/files in the repository. To begin, you need to clone the remote repository to a local repository on a machine. For example, to clone our class repository to a ugrad server:

    1. ssh to ugrad and type cd ~ to make sure you are in your home directory.
    2. Run
      git clone https://github.com/jhu-ip/cs220-summer21-public.git
    3. Type ls to check what's in your home directory. You should see a new directory called cs220-summer21-public has been created. This is the local repository visualized using the OS's file system, which is a copy of the remote repository. Now you can navigate the repository using Linux commands.

  3. Clone your personal repository similar to how you have cloned cs220-summer21-public.

  4. Now, let's rename your local personal repository directory to cs220: mv ~/2021-summer-student-JHED ~/cs220 and also rename your local course repository to cs220-public From now on, we will refer cs220 as your personal local repository and cs220-public as the course local repository. If you name it differently, don't be confused by what cs220 and cs220-public directories are.

Tip
  1. Do not use the ssh version of the clone command; use the https one.
  2. If you already have the repository cloned, you can update the local repository by pulling the changes from the remote repository using the command git pull.
Checkpoints#

Now you should have successfully cloned remote repositories to the ugrad machine. Here are some checkpoints you should ask yourself:

  1. What is a remote repository?
  2. How do you clone a remote repository?
  3. What is a local repository?
tip

If you did not successfully complete the above steps, ask a member of the course staff and/or post on Piazza.

Make changes in a local repository#

The first thing you want to do is to create a README for your personal repository.

  1. Use cd to move into cs220 directory. Use pwd to check that you are in the right place.

  2. Use a text editor (emacs or vim) to create a new text file called README there. Put your name, JHED, class year, and major(s)/minor(s) in your README.

  3. Type the command git status to see what it reports. You should see all potential changes in your local repository, which is the README file and it said it is un-tracked.

  4. You created a potential change with the new README file. However, this change is not added to your local repository yet. To add it to your local repository, you need to use the command git add REAMDE.

  5. Type the command git status to see what it reports. You should see what has been added in your local repository, which is the README file and it said it is added.

  6. Up to now, the change has been added to the local repository but not applied yet. To apply the changes, you need to commit it using git commit -m "created README".

  7. Because it is your first commit, git likely asks for your name and email. You may see something like this:

    *** Please tell me who you are
    Run
    git config --global user.email "you@example.com"
    git config --global user.name "Your name"
    to set your account's default identity.
    Omit --global to set the identity only in this repository.

    Follow these instructions and set up the global committer identity and commit the change to your local repository.

    How do I commit with my identity to someone's local repository temporarily?

    If you work in a team, you may be working on your teammate's machine directly. After you make changes in the local repository of your teammate, you may want to commit the changes and push them to the remote repository with you as the committer. In this case, you may want to run this commit command to specify the committer's name and email:

    git -c user.name="Your name" -c user.email="you@example.com" commit -m "messages"

  8. Now, type the command git status to see what it reports. You should see nothing has been reported, i.e. no more potential changes in your local repository.

  9. Type the command git log to see what it reports. You should see a new entry with the commit message "created README".

  10. Edit your README file, and within the file, add a line saying that this is your personal repository for Intermediate Programming at JHU for Summer 2021.

  11. Check the status again with git status, then add and commit your updated README using git commit -am "updated README".

    info

    The -a flag commits all changed files. It is convenient to use it if you are sure that you want to add all the potential changes.

  12. Type the command git log to see what it reports. You should now see two entries.

Checkpoints#
  1. How do you add a change to the local repository?
  2. How do you add all changes to the local repository?
  3. How do you confirm/commit the added changes to the local repository?
  4. Look it up: how do you reset the added changes before you commit?
  5. Look it up: how do you revert the last commit?
  6. How do you specify the committer identify for a specific commit (instead of using the global git configuration)?
Push changes to the remote repository#

So far, we made changes in the local repository (by git add and git commit). You will need to push these changes to the remote repository.

  1. Push changes to the remote repository using the command git push.
  2. Now, use the web browser and go to your personal repository on Github. You should see all the changes you have made locally have been reflected in the remote repository.
Checkpoints#
  1. If your local repository is not up-to-date, can you still git push the local changes to remote?

Part 7 - Coding assignment submission workflow#

Introduction#

In this course, you will be submitting your assignments to Gradescope. In this part, you will use ex2 to prepare an assignment submission - consider this a trial run for turning in an assignment later on.

Instructions#

Suppose that an assignment asks you to turn in your codes, a new README file, along with a copy of your git log, to show how you have been using git during the assignment.

  1. Save a copy of your git log to a text file by typing the command git log > gitlog.txt.

  2. Use zip as indicated below to bundle/compress your repository into a .zip file named ex2.zip. The command should look something like: zip ex2.zip README gitlog.txt. Note that the first argument after zip is the name of the .zip file to create; the remaining arguments are the files or folders to bundle together. -r flag may be needed if you want to zip the content recursively.

  3. Use cd ~ or just cd to move into your home directory.

  4. Now move (the command mv means to move, rather than copy) your .zip file to your current location using the following command: mv cs220/ex2.zip .. Note that the . in the command above indicates that the desired destination is your current working directory.

  5. Check to make sure your .zip file contains the right files with unzip -l (this says to list the files that would come out if you were to unzip the bundle) as indicated below: unzip -l ex2.zip. This will cause a list of the files and directories in your zip file to be displayed on the screen. It is your chance to double-check that you have included everything. If you wanted to actually unzip your bundled files (though you do not need to do so now), you could use the unzip command as above, but without the -l flag. You will often want to unzip your file in a folder which is different than the one where you created it, to avoid overwriting your original README and gitlog.txt files.

  6. Now that all required files are bundled together as one .zip file, you will need to copy the bundle from ugrad to your local machine. Follow Part 3 for detailed instructions.

  7. If you wanted to submit this on Gradescope, you now have a copy of your .zip file on your local computer, and could launch a web browser that will allow you to select the file you wish to upload to Gradescope.

File transfer via Git

Alternatively, after you have pushed all local changes to the remote repository on your local machine, you can clone the remote repository or pull to update the local repository, then you can directly zip the files that you want to submit. However, if you work locally on your local machine from time to time and only transfer files to ugrad server for testing, you should avoid using Git as the bridge of transfer during debugging. In the past, students doing in this way generated hundreds of meaningless commit messages, which resulted in a point deduction. You should not commit while you are still debugging. A commit should reflect meaningful progress.

Checkpoints#

  1. How do you put your git commit history into a file?

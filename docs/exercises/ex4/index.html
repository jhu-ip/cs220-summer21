Day 4 In-class Exercise

info

This is an in-class exercise. An exercise page like this one will contain a brief description but is intended to be supplemented by discussion during our meeting time. Complete the exercise to the best of your ability. Feel free to talk with other students as you work, and do not be afraid to ask questions if you get stuck. Aim to complete as much as possible during our meeting, and try to finish on your own before our next class meeting. Remember to submit on Gradescope once or more, checking the autograder results. For credit, you must submit by the end of the semester.

Learning Objectives#

Objectives

This exercise will help you gain additional familiarity with:

  • Function declarations
  • Using sizeof
  • Passing arrays to functions
  • Recursion
  • Modularizing your code by writing separate functions
  • Organizing functions into separate files (function declarations in header files)
  • Using header guards
  • Using separate commands for compilation (translation to object code) and linking
  • Creating a Makefile and using make to build your program

Part 0 - Get the starter codes#

note

Remember that in previous exercises; we renamed your copy of the course repository and your personal repository to cs220-public and cs220 for brevity. If this is not the case, ask for assistance. You may need to redo steps from earlier exercises.

To get the start codes, you should:

  1. ssh to a ugrad server.
  2. Navigate to cs220-public.
  3. Type git status and confirm you have not modified any files or accidentally committed to the course repository. That is, you should not see any files categorized as "Changes to be committed", "Changes not staged for commit" or "Untracked." You should also not see the message "Your branch is ahead of 'origin/master' by X commit(s)." (If you do, please speak with an instructor or CA for a fix.)
  4. Type git pull to synchronize cs220-public with the remote repo.
  5. Confirm that you can see the starter codes for today’s exercise by typing ls exercises/ex4 -- you should see files listed below:
    ~/cs220-public/exercises
    └── /ex4
    ├── functions.c
    ├── all_in_one.c
    └── Makefile
  6. Now, navigate to your cs220/exercises folder. (The exercises directory that you have created in previous exercises)
  7. Make a new directory for today's exercise. e.g. mkdir ex4.
  8. Copy the starter codes for today's exercise from cs220-public to the directory you just created. e.g. cp ~/cs220-public/exercises/ex4/* ~/cs220/exercises/ex4/.
  9. Navigate to this new directory and verify all the starter codes are there.
  10. Add and commit the changes to your local repository, e.g. git add ., then git commit -m "Added Ex4 starter codes".
  11. Push changes to the remote repository. You should always pull it to confirm that your local repository is up-to-date before pushing changes. e.g. git pull, then git push. If you encounter difficulties, do not hesitate to ask for assistance.

Checkpoints#

From now on, you should be familiar with these steps to get the start codes from cs220-public, copy them over to cs220, and create an initial commit.

Part 1 - Function declaration#

Introduction#

In this part, you will write the function declaration for div in the given starter code functions.c. You will learn when you need to write a function declaration.

Instructions#

  1. Compile your newly-copied functions.c file by typing our usual compile command: gcc -std=c99 -Wall -Wextra -pedantic functions.c (or use gccc functions.c if you set up the alias during ex3). You will notice that the program cannot be compiled because the div function is called before it is defined.

  2. Edit functions.c to include a declaration of the div function before main, similar to examples in the lecture slides.

  3. Once you have made your changes, re-compile the file by re-typing the compile command: gcc -std=c99 -Wall -Wextra -pedantic functions.c (or use gccc functions.c.)

  4. Test your program by executing ./a.out and reviewing the output.

Checkpoints#

  1. Why do we need to declare div before main?
  2. What if we move the definition of div before main? Do we still need to declare it now? Why or why not?
  3. Can we declare the same function multiple times? Try it out.

Part 2 - Writing functions#

Introduction#

Now, you will practice how to write functions in C. In this part, you will write new functions called mult and fac in functions.c.

Instructions#

  1. Write code to declare and define a mult function, which takes two floating-point values and returns the result of their product. The function should have a prototype like float mult(float a, float b).

  2. Next, implement a recursive fac function, which takes a non-negative integer and returns the factorial of it. If the input is negative, return 0. The function should have a prototype like int fac(int a) or maybe long fac(int a) since factorials get large very quickly.

  3. Uncomment the corresponding test lines in main in functions.c, compile and run the program to test your implementations.

Checkpoints#

  1. How did you declare and define the functions? Did you declare and define it at the same time or separately?
  2. How did you implement fac? Did you use recursive calls?
  3. If you did not implement fac recursively, try it out.
  4. What does %ld used in the printf format string mean?

Part 3 - Binary search (Recursion)#

Introduction#

Binary search is an algorithm to search for a target in a sorted array and return its position. It can be implemented using recursion. In this part, you will implement a binary search on an array of floats using recursion.

Instructions#

  1. Read this page binary search to get familiar with the algorithm.

  2. Still working in the functions.c file, write a function declaration and definition to do a (recursive) binary search on an array of floats. The prototype of your method should be int bsearch(float ra[], int low, int high, float target) where low is the inclusive low index of the range to search, high is the inclusive high index of the range to search, target is the value to find, and the function returns the location (index) of the target in array ra, or -1 if it is not found.

  3. Uncomment the corresponding test lines in main in functions.c.

  4. Compile and run your updated program to see if it works properly. Fix any errors you encounter.

  5. Add another function with this prototype int bsearch2(float ra[], int low, int high, float target, float results[], int size). The extra parameter results is an array of floats that holds the values inspected during the search for the target, and size is a variable that keeps track of how many values have been added to results.

  6. Uncomment the corresponding test lines in main in functions.c, compile and run the program to test your implementations.

Checkpoints#

  1. What is the base case/stopping criteria of your recursive calls?
  2. What is the difference between passing an int to a function and passing an array of int to a function?
  3. Why can you print the updated results array in main, but not get the updated number of values added to it?

Part 4 - C string concatenation#

Introduction#

In this part, you are going to work on all_in_one.c. This program reads in two string inputs and concatenates them. You are going to compile, debug, and finish the concatenation implementation.

Instructions#

  1. Compile your newly-copied all_in_one.c file by typing our usual compile command: gcc -std=c99 -Wall -Wextra -pedantic all_in_one.c (or use gccc all_in_one.c if you like). You will notice that the function calls do not return anything sensible at the moment, so let's fix that.

  2. Edit all_in_one.c so that the function named concat performs string concatenation of its first two arguments, “returning” the concatenated string via its third argument and sending 0 back as the function's return value. However, note that the function's third argument has limited capacity, the value of which is indicated in the fourth argument. If concatenating the first two arguments would cause an overflow of the third argument, do not modify the third argument and return the value 1 to indicate an error situation.

    For example, if the first two arguments are "alpha" and "beta", and the result string has capacity 10, then the value returned by the function will be 0 and, after the function call, the value inside the third argument should be "alphabeta". However, if the first two arguments are "alpha" and "gamma", with the fourth argument of 10, the value returned by the function will be 1, and the value inside the third argument will not be changed because concatenating them would not leave any room for the null string terminator character.

caution
  • You may NOT use the strcat or strcpy library functions (from string.h) in your solution. But you MAY use strlen.
  • You should NOT modify any code in the main function. Edit only the concat function, and get it to work with the existing function calls in main. At that point, if you would like to test your function further, you may add additional calls in main to do so.

  1. Once you have made your changes, re-compile the file by re-typing our usual compile command: gcc -std=c99 -Wall -Wextra -pedantic all_in_one.c (or use gccc all_in_one.c.)

  2. Test your program by executing ./a.out and reviewing the output. It is okay to add new calls in main if you wish.

Checkpoints#

  1. How do you determine the length of a C string?
  2. What is a null-terminator? What happens if the null-terminator is missing in a string?
  3. What happens if you access an array with an out-of-bound index? (e.g. int a[10];, and you access it with a[20].)

Part 5 - Separate compilation#

Introduction#

In this part, you will split your working source code into multiple files and compile them separately. Then, you link all the object files to create the executable.

Instructions#

Without changing all_in_one.c, copy the contents of your all_in_one.c program into the following files:

  1. run_concat.c - should contain the main() function, but no other functions.

  2. string_functions.c - should contain the definition for the string helper function concat (and in theory, could later include additional function definitions). It should not have a main() function.

  3. string_functions.h - should contain a prototype (i.e. declaration) for each function defined in string_functions.c (today, there is only one), and should be #include-d in both of the .c files. Make sure to include a header guard for this file to prevent being included multiple times.

Compile and link the program manually, step by step.

  1. Recall that to compile a source file only (and not link or generate an executable), you will use the regular gcc command with our usual flags and the -c flag with the .c source file as an argument. This command will create an object file.

  2. Once you have created your two object files, you can link them to create an executable named run_concat by using the gcc command with both .o files as arguments (but here, you do not need the usual flags we use for compiling). To express in the command that you want the executable file named run_concat, you will need to use the -o flag with run_concat as an argument.

  3. If your program does not compile or link properly, check that you have added the necessary #include statements in each .c file.

  4. Do not move onto the next step until you can successfully build and run the run_concat executable file.

Checkpoints#

  1. What is the difference between compilation and linking?
  2. Why do we need header guards in the header files?
  3. Is it a good idea to implement the functions in the header files instead of source files? Why or why not?

Part 6 - Makefile#

Introduction#

In this part, you will create a Makefile for this program so that you can build it quickly and easily.

Instructions#

  1. Now it is time to edit the sample Makefile file you copied over from the public repository back in Part 0.

  2. Modify Makefile so that it builds the run_concat program and creates an executable called run_concat (rather than main).

  3. Try out the following command-line commands and try to reason out what is happening at each step by reading the messages it prints: make, make clean, make string_functions.o, make run_concat.o. Look around in the folder using ls -l after each command to see what changed. Execute them in different orders (and after modifying some source file slightly) and see how the behavior changes.

  4. Finally, check that your Makefile captures the required dependencies properly:

    • Edit the string_functions.c file so that the concat function behaves noticeably differently, e.g., outputs a message on the screen. (It does not matter what it says.) Save it, but do not re-compile it.
    • At the command line, type make. Then execute your program and confirm that the output you see is due to the new behavior of the function.

Checkpoints#

  1. Why do we use Makefile?
  2. What is a target defined in a Makefile?
  3. How do you specify the dependencies between source files and a target?

Part 7 - Gradescope submission#

Create a zip file called ex4.zip, which contains functions.c, all_in_one.c, run_concat.c, string_functions.[c|h], and Makefile with your implementation. Submit the zip file as Day 4 In-class Exercise on Gradescope. Make sure your program passes all the autograder tests in order to get full credit. It is due before the next class. However, late submission is allowed until our last class. Even so, we highly recommend you to keep up the schedule, finish each in-class exercise, and submit it on Gradescope before the next class.
